一、CommonUI 概述

CommonUI 是 Epic Games 在 UE5 中正式推出的跨平台 UI 开发框架，其前身最早可追溯至 UE4.27 的 Experimental 阶段。官方在开发《Fortnite》跨平台版本时深刻体会到：一套代码同时适配键盘鼠标、手柄和触屏三种输入模式所带来的复杂度呈指数增长。CommonUI 正是这一经验的产物。

1.1 解决的核心问题

在传统 UMG 开发流程中，开发者需要手动处理以下问题：

  • 输入模式切换：键盘鼠标与手柄之间的光标显隐、导航方式切换
  • 按钮状态管理：Hover / Press / Focus / Disabled 等状态的样式和音效
  • 焦点导航冲突：手柄 D-Pad 导航与鼠标点击的焦点抢占
  • 跨平台样式：同一套 UI 在 PC 和主机上的视觉差异处理
  • 激活栈管理：多层弹窗时的输入归属和模态控制

CommonUI 将这些共性需求抽象为统一的基类和配置资产，开发者只需继承和配置，无需从零搭建。其设计哲学可概括为三条：输入无关（Input Agnostic）、平台无关（Platform Agnostic）、样式数据驱动（Style Data-Driven）。

1.2 与普通 UMG 的关系

CommonUI 并非替代 UMG，而是建立在 UMG 之上的一层高级封装。所有 CommonUI Widget 仍然是 UUserWidget 的子类，兼容 UMG 的全部功能。两者的关系类似 UE 的 APawn 与 ACharacter：后者在前者基础上内置了移动组件和动画蓝图等常用能力，CommonUI 则为 UMG 内置了输入路由、焦点管理和样式系统。

二、核心架构

2.1 ActivatableWidget — 最小可激活单元

UCommonActivatableWidget 是 CommonUI 的基石。任何需要独立管理输入和焦点的 UI 面板都应继承它，而非直接继承 UUserWidget。

核心生命周期：

阶段 方法 说明
构建 NativeOnInitialized() Widget 首次初始化
激活前 NativeOnActivated() 即将获得输入焦点
激活后 NativeOnActivated() 返回后 已接收输入路由
去激活前 NativeOnDeactivated() 即将失去输入焦点
激活状态变化 OnActivationStateChanged() 状态机回调

UCommonActivatableWidget 内置了 bIsTransitioning 防止重入激活，以及 GetDesiredInputMode() 让每个面板声明自己期望的输入模式。

2.2 CommonButtonBase — 跨平台按钮

UCommonButtonBase 是 CommonUI 中最重要的交互控件。它通过三层抽象处理跨平台交互：

  1. 交互方法（Interaction Method）：自动检测当前输入类型（鼠标/手柄/触摸），切换对应的样式和行为
  2. 选择状态（Selection State）：Pressed / Released / Hovered / Selected / Disabled
  3. 触发动作（Triggering Action）：通过绑定 DataAsset 而非硬编码按键

关键配置属性：

  • bSelectable — 是否可通过手柄导航选中
  • InputActionData — 关联的 CommonInputActionDataBase
  • TriggeringEnhancedInputAction — 绑定的 Enhanced Input Action

2.3 CommonTabListWidgetBase — Tab 切换容器

UCommonTabListWidgetBase 提供了一套完整的 Tab 页签管理方案。它自动处理：

  • Tab 注册与注销
  • 选中状态同步
  • 上一个/下一个 Tab 导航
  • 关联内容面板的切换

Tab 切换时，CommonTabListWidgetBase 自动调用前一个面板的 DeactivateWidget() 和新面板的 ActivateWidget()，开发者无需手动管理焦点转移。

三、输入模式系统

3.1 三种 InputMode

CommonUI 定义了三种输入模式，由 ECommonInputMode 枚举管理：

模式 光标 手柄焦点 适用场景
Menu 可见 强制显示 主菜单、设置界面
Game 隐藏 不拦截 游玩中 HUD
All 半透明 叠加显示 背包半透明覆盖

输入模式的切换由 UCommonUIActionRouterBase 统一管理。当 ActivatableWidget 被激活时，系统对比其 GetDesiredInputMode() 与当前模式，若不同则自动切换。切换过程有平滑过渡（InputModeTransition），避免视觉闪烁。

四、聚焦管理

4.1 Focus Path 机制

CommonUI 的焦点管理基于 Slate 的 Focus Path，但增加了平台感知层。核心类 UCommonUIActionRouterBase 维护一棵激活树，确保焦点始终落在当前激活栈最顶层的 Widget 上。

焦点流程：

  1. 新 Widget 激活 → 注册到激活树
  2. 系统遍历 Focus Path → 确定下一个应该获得焦点的控件
  3. 若当前输入设备是手柄 → 自动将焦点导航到第一个 bSelectable 控件
  4. 弹窗关闭 → 焦点恢复到上层 Widget 的 “挂起焦点”

4.2 导航策略

Explicit 手动设置每个方向的导航目标 不规则布局
Custom 通过 OnNavigation 事件自定义逻辑 环形菜单
CustomBoundary 自定义 + 边界溢出处理 网格布局

十、常见坑点与最佳实践

10.1 焦点丢失问题

现象：打开弹窗后，手柄导航焦点消失，D-Pad 无法选中任何按钮。

原因：激活栈中的新 Widget 没有可聚焦控件，或 NativeOnActivated 中未手动调用 SetFocus()。

10.2 Enhanced Input 冲突

现象：UI 打开后，Enhanced Input 的 Triggered 事件同时触发 UI 和游戏逻辑。

原因：Enhanced Input 的 Priority 未正确设置，UI 输入未被提升。

解决：在 GetDesiredInputConfig() 中返回 Menu 模式并配合 Enhanced Input 的 Priority 设置：

10.3 CommonTextBlock 样式不生效

现象：设置了 CommonTextStyle DataAsset 但文本外观无变化。

原因：Widget 类型是普通 UTextBlock 而非 UCommonTextBlock。CommonUI 的样式系统仅对 Common 前缀的控件生效。

解决：将所有文本控件改为 CommonTextBlock，按钮改为 CommonButtonBase。

10.4 激活栈内存泄漏

现象：反复开关 UI 后内存持续增长。

原因：动态创建的 ActivatableWidget 未正确释放，或激活栈中存在循环引用。

最佳实践：

  • 非持久性弹窗在 NativeOnDeactivated 中调用 RemoveFromParent()
  • 使用 UCommonActivatableWidgetStack 管理弹窗生命周期
  • 避免在 NativeOnActivated 中创建大量动态 Widget（改用对象池）

10.5 平台检测时序问题

现象：HasTrait() 在 NativeOnInitialized 中返回错误值。

原因：UCommonUISubsystem 的平台特性检测依赖于输入设备检测，而设备检测在 Widget 初始化时可能尚未完成。

解决：将平台判断逻辑放在 NativeOnActivated() 中，而非 NativeOnInitialized()。

10.6 最佳实践总结

  1. 始终使用 Common 前缀控件：CommonTextBlock、CommonButtonBase、CommonActivatableWidget 等
  2. 样式走 DataAsset：不要硬编码颜色/字体，统一走 CommonTextStyle / CommonButtonStyle
  3. 输入走 CommonInputActionData：不要直接绑定按键，让 CommonUI 做跨平台映射
  4. 激活栈统一管理：复杂 UI 场景使用 UCommonActivatableWidgetStack
  5. 测试三平台输入：键盘鼠标、手柄、触摸都测一遍，尤其关注焦点切换
  6. 参考 Lyra：遇到设计困惑时，先在 Lyra 项目中搜索类似场景的实现
