一、概述：GoF 设计模式与游戏开发的交汇

1994 年，四人组（Gang of Four）出版的《设计模式：可复用面向对象软件的基础》定义了 23 种经典模式，分为三大类：

分类 核心理念 模式
创建型 将对象创建与使用解耦 单例、工厂方法、抽象工厂、建造者、原型、对象池
结构型 通过组合构建更大结构 适配器、桥接、组合、装饰器、外观、享元、代理
行为型 对象间职责分配与算法封装 观察者、策略、状态、命令、模板方法、迭代器、责任链、中介者、备忘录、访问者、解释器

游戏开发对设计模式的诉求与常规企业应用有本质不同：

  1. 性能敏感：每帧 16ms（60FPS）的预算下，对象的创建/销毁成本不能忽视。
  2. 实体复杂度高：一个 BOSS 可能同时拥有 60+ 种 Buff 效果、20 段行为树、3 个阶段状态。
  3. 状态多变：从主菜单到关卡内到结算界面，每一帧的上下文都在切换。
  4. 多用组合少用继承：传统 OOP 的深继承链在游戏实体上容易形成「钻石继承」灾难。

这些特殊性使得部分 GoF 模式在游戏开发中「身价倍增」（如组件模式、状态模式），而另一些则很少用到（如解释器模式）。

二、创建型模式

2.1 工厂模式：AI 与技能的动态创建

工厂模式隐藏对象创建细节，让调用者只依赖抽象接口。在游戏开发中，工厂模式最常见的三类应用：

技能工厂

优点在于：新增技能只需修改工厂内部 switch，调用方代码零改动。如果配合 DataTable 驱动（UE 中每个技能一行配置），甚至连工厂代码都无需修改，真正实现数据驱动的技能系统。

AI 行为工厂

敌人波次工厂

将敌人生成逻辑封装为 EnemyWaveFactory，通过 JSON/CSV 配置文件定义每一波敌人的组合、刷新间隔、难度系数，运行时由工厂解析并生成实体。

2.2 对象池模式：高频创建的性能救星

游戏中最频繁的创建/销毁操作集中在：子弹、粒子特效、音效实例、场景中的小道具。如果每发子弹都用 new/delete（C++）或 Instantiate/Destroy（Unity），GC 压力和内存碎片会迅速拖垮帧率。

对象池的核心机制：预创建 + 激活/休眠 + 复用。

实现要点：

  • 为每种频繁创建的类型维护独立池（避免类型转换开销）。
  • 池耗尽时的策略：扩容（动态分配）或复用最早激活的（LRU）。
  • UE 中 UObjectPool 和 UActorPool 已内置此模式，应优先使用引擎方案而非手写。

2.3 单例模式：GameInstance 的正确与错误用法

单例是游戏开发中最常用的模式，也是最容易被滥用的模式。

正确的使用场景：

引擎 单例载体 用途
Unity [RuntimeInitializeOnLoadMethod] + 静态类 全局音频管理器、存档管理
Unreal UGameInstanceSubsystem 全局配置、成就追踪、跨关卡数据
Unreal UWorldSubsystem 世界级服务（AI 管理器、环境查询）

滥用警告：

单例的代价是全局状态，在多人游戏中尤为致命。UE 的 Subsystem 体系提供了生命周期管理（随 World 销毁而清理），比裸单例安全得多。

三、结构型模式

3.1 组件模式：UE Actor-Component 的精髓

组件模式是游戏开发中最重要的结构型模式，甚至超越了 GoF 的原始定义。核心思想：实体通过组合多个组件定义行为，而非通过继承链。

UE 的 Actor-Component 架构是组件模式的典范：

对比继承方式：

组件模式解决了三大痛点：钻石继承、代码复用困难、运行时动态组合。

设计要点：

  • 每个 Component 只负责一个清晰的职责（单一职责原则）。
  • Component 之间通过接口通信而非直接引用（如 UHealthComponent 暴露 OnDamaged 委托）。
  • 避免 Component 的循环依赖——如有必要，引入第三个 Component 做协调。

3.2 装饰器模式：Buff/Debuff 的动态叠加

RPG 和 MOBA 中的 Buff 系统是装饰器模式的经典应用。一个角色可能同时受到：攻击力提升 30%、移速降低 20%、每秒回血 50 点、受到伤害时反弹 15%。这些效果需要动态叠加和撤销，且彼此独立。

UE 中的实现建议：Buff 系统的 Decorator 链不适合在 Tick 中全量重新计算（性能开销大），应维护一个 Dirty Flag——仅在有 Buff 添加/移除时重新计算，并将结果缓存。

3.3 外观模式：复杂子系统的简洁入口

游戏引擎的底层子系统极其复杂（渲染管线、网络同步、音频混音、物理模拟），外观模式将它们封装为高层 API，降低业务逻辑层的认知负担。

4.3 命令模式：输入缓冲与回放系统

命令模式将请求封装为对象，使请求的发送者与执行者解耦。在游戏开发中的三大应用：

输入缓冲

格斗游戏中的「预输入」系统：玩家在攻击动画播放期间按下后续按键，命令被缓存并在当前动作结束后执行：

技能队列

MMORPG 中玩家按下技能热键，技能命令进入队列，GCD（全局冷却）结束后按 FIFO 顺序执行。

回放系统

将每一帧的输入命令序列化为时间线，回放时按时间戳重新执行命令队列即可完美复现对局。RTS 和 MOBA 的录像系统均基于此原理。

4.4 策略模式：AI 行为的动态切换

AI 的行为不应硬编码在 AAIController 中。策略模式将算法（行为策略）封装为可互换的对象：

策略切换时机：由 Blackboard（行为树感知数据）或感官系统（视觉锥、听觉范围）触发策略替换。策略模式与状态模式的不同在于：状态模式关注「角色当前在做什么（状态）」，策略模式关注「如何做（算法）」。

五、Game Programming Patterns 补充

Robert Nystrom 在《游戏编程模式》中补充了几种游戏专属模式，与 GoF 模式互补：

模式 核心思想 与传统模式对比
Game Loop 固定时间步长的主循环驱动游戏世界 无对应 GoF 模式，是游戏架构的基础
Update Method 每个实体每帧调用 Update() 模板方法模式的游戏化变体
双缓冲 渲染到后台缓冲，完成后交换 代理/装饰器模式的特化应用
空间分区 按空间区域管理实体，减少 O(n²) 查询 组合模式 + 策略模式的空间化应用
脏标记 仅在需要时重新计算缓存值 观察者模式的性能优化变体

这些模式强调了游戏开发的特殊性：帧率约束下的计算效率优先。传统 GoF 模式需要根据这一约束进行「游戏化」适配（如装饰器配合脏标记避免每帧重新计算）。

六、UE 内置模式实现

Unreal Engine 自身大量使用了设计模式，理解这些内置实现有助于写出更「UE 风格」的代码：

UE 机制 对应模式 说明
UGameInstanceSubsystem / UWorldSubsystem 单例 + 服务定位器 框架管理生命周期，比裸单例安全
UActorComponent 组件模式 Actor-Component 是组件模式在游戏领域的巅峰实现
GC（垃圾回收） 享元模式 TArray 元素通过索引访问而非指针，减少内存碎片
FTickerDelegate 观察者模式 每帧 Tick 通过委托广播，支持优先级排序
UnrealBuildTool 建造者模式 ModuleRules 构建编译配置，最终由 Builder 生成项目文件
FObjectInitializer 原型模式 通过 CDO（Class Default Object）克隆生成新实例
TWeakObjectPtr 代理模式 代理 GC 管理的指针，安全访问可能已被销毁的对象
DataAsset / DataTable 享元模式 同一数据资产被数百个实例共享
FSlateApplication 外观模式 统一入口管理输入、渲染、窗口

八、常见反模式与过度设计陷阱

8.1 反模式

反模式 症状 修复
上帝单例 GameManager 有 3000+ 行，引用一切 拆分为 Subsystem / Service
链式策略地狱 每个小差异都新建策略类，最终 50+ 策略 用参数化替代策略爆炸
装饰器过深 Buff 装饰链超过 10 层，每帧计算 100+ 次 脏标记 + 缓存 + 合并同类 Buff
观察者泄漏 委托未解绑，对象销毁后仍收到广播 使用 Weak Lambda / 在 EndPlay 中 RemoveAll
状态爆炸 状态数超过 30 个，转换矩阵 N² 规模 引入层级状态机（HFSM）或行为树
过度工厂 为 3 种子弹创建 AbstractFactory + ConcreteFactory × 3 一个带参数的静态工厂方法足矣

8.2 过度设计陷阱

  1. 「所有类都必须有接口」：如果只有一个实现类，接口就是多余的抽象。等第二个实现出现时再提取接口（YAGNI 原则）。
  2. 为简单逻辑引入命令模式：如果操作只是 player.Jump()，不需要包装成 JumpCommand。命令模式的价值在队列/撤销/序列化场景才体现。
  3. 强行套用 23 种模式：实际项目中使用 8-10 种核心模式已足够。组件模式、状态模式、观察者模式、策略模式、命令模式、工厂模式、对象池、单例（Subsystem）覆盖了 90% 的游戏开发需求。
  4. 忽略引擎内置方案：UE 的 Subsystem、GameplayAbilitySystem、DataAsset 已经内置了大量设计模式。在造轮子之前先查引擎文档。

设计模式的精髓不在记住 23 个名字，而在于理解每个模式背后的设计原则：单一职责、开闭原则、依赖倒置、接口隔离。当你面对一个设计问题时，能从这些原则出发推导出模式化的解决方案，才是真正掌握了设计模式。
