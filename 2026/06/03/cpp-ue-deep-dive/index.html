C++ 深度解析 - 现代特性与 UE 引擎开发实战

1. C++ 在 UE 中的角色

UE5 的核心由约 400 万行 C++ 代码构成。从渲染管线（RHI → RDG → 平台后端）到网络同步（Replication Graph）、从 GAS（Gameplay Ability System）到 Niagara VFX、从 Chaos 物理到 MassEntity 框架——所有性能敏感路径都由 C++ 实现。

蓝图很好，但在以下场景必须回归 C++：

  • 每帧执行的热路径：Tick 中高频调用的逻辑，蓝图虚拟机开销不可忽略。
  • 大规模数据操作：数千个 Actor 的批量处理，C++ 的数据局部性优势碾压蓝图。
  • 底层系统对接：自定义物理查询、渲染 Hook、网络序列化等只有 C++ 能触及。
  • 第三方库集成：Protobuf、RPC 框架、加密库等需要 C++ 桥接。

理解 UE C++ 不仅仅是掌握语言本身，更是理解 UE 在标准 C++ 之上构建的一整套反射、GC、序列化、委托体系。

2. 现代 C++ 特性与 UE 的结合

2.1 智能指针

UE 实现了自己的一套智能指针体系，与标准库并行存在：

UE 类型 std 对应 用途
TSharedPtr<T> std::shared_ptr<T> 共享所有权，非 UObject
TUniquePtr<T> std::unique_ptr<T> 独占所有权
TWeakPtr<T> std::weak_ptr<T> 弱引用，配合 TSharedPtr
TSharedRef<T> 无直接对应 非空共享引用
TObjectPtr<T> 无直接对应 UObject 专用，UE5 引入

核心规则：UObject 及其派生类禁止使用 TSharedPtr，必须走 UE GC 体系。UE5 引入 TObjectPtr<T> 替代原始 UObject*，在编辑器模式下提供访问追踪和懒加载支持。

2.2 移动语义

UE 的容器（TArray、TMap、TSet、FString）全面支持移动语义，关键规则：

  • 返回局部变量自动触发移动（NRVO / 隐式移动）。
  • MoveTemp() 是 UE 的 std::move()，显式转移所有权。
  • TArray::Emplace() 原地构造，避免临时对象的拷贝+析构。

2.3 Lambda 与 TFunction / TDelegate

类型 特点
TFunction<Signature> 通用可调用对象包装，类似 std::function，支持 lambda/函数指针/仿函数
TDelegate<Signature> UE 委托，可绑定 UFUNCTION，支持序列化、网络复制
DECLARE_DELEGATE / DECLARE_MULTICAST 宏声明委托类型

陷阱：AddLambda 返回的 handle 无法用于 Remove，只能用 RemoveAll(this) 批量清除。建议用 AddUObject 绑定 UFUNCTION 以获得精确的移除控制。

3. UE 反射系统

3.1 UHT 原理

UE 的反射不是标准 C++ RTTI，而是通过 UHT（Unreal Header Tool）实现的代码生成系统。流程：

  1. 解析 .h 文件中的 UCLASS/UPROPERTY/UFUNCTION 等宏标记。
  2. 为每个标记生成 *.generated.h 和 *.gen.cpp 文件，内含该类的 UClass 元数据。
  3. 编译时注入到可执行文件，运行时通过 StaticClass() 访问。

3.2 宏展开与 GC 标记

GC 标记阶段从根集出发，通过 UPROPERTY 找到所有引用，递归标记可达对象。未被标记的 UObject 在清除阶段被回收。任何未用 UPROPERTY 标记的 UObject 指针都是 GC 不安全的——这是 UE C++ 的「新手杀手」。

3.3 TObjectPtr vs 裸指针

UE5 引入 TObjectPtr<T> 替代原始 UObject*：

  • 编辑器下提供访问追踪，帮助发现 GC 问题。
  • 支持异步加载时的占位机制。
  • 未来可能实现更高效的内存布局（如压缩指针）。

4.2 UObject GC

触发时机：UWorld::Tick 中的 ConditionalCollectGarbage()、关卡切换、手动调用 GEngine->ForceGarbageCollection()。

GC 是分帧增量执行的：每帧执行固定时长的标记/清除步骤，避免单帧卡顿。大世界场景中尤其重要——一次性遍历所有 Actor 引用可能耗时数百毫秒。

优化建议：

  • 减少 UObject 总数量：合并不需要独立 GC 跟踪的小对象。
  • 避免 AddToRoot()：Root 集中的对象永不回收，除非显式 RemoveFromRoot()。
  • 使用 FGCObjectScopeGuard：RAII 风格保护临时 UObject。

5.1 三层并发模型

层级 组件 适用场景

5.2 线程安全容器

容器 特点
TQueue<T, EQueueMode::Mpsc> 多生产者单消费者无锁队列

FCriticalSection 传统互斥锁

Golden Rule：切勿在持有锁的状态下调 UE 引擎 API（可能触发 GC、资源加载、网络同步等导致死锁）。

6.3 虚拟函数开销

虚函数调用有两个开销：vtable 间接跳转（~2ns）+ 阻止内联优化。在每帧百万次调用的热路径上，虚函数开销会累积。

替代方案：

  • CRTP（奇异递归模板模式）：编译期多态，零运行时开销。
  • final 关键字：标记 virtual void Tick(float) final，编译器可去虚拟化。
  • 数据导向设计（DOD）：按类型分组批量处理，而不是逐个调用虚函数。

7. 实战案例：AOI 系统

AOI（Area of Interest）是 MMO/大世界游戏的核心系统，决定每个玩家能看到哪些其他实体。

设计要点：

  • 空间哈希分桶，查询复杂度 O(可视格子数) 而非 O(总实体数)。
  • 细粒度锁（每个 Cell 独立锁）降低争用，ParallelFor 并发查询。
  • TUniquePtr<FCell> 确保所有权清晰，Grid 调整时自动析构。

8.2 跨模块依赖

优先使用 PrivateDependency，减少模块间耦合。避免循环依赖（A 依赖 B 同时 B 依赖 A），必要时提取公共接口到第三个模块。

8.3 热重载限制

Live Coding（Ctrl+Alt+F11）的限制：

  • 不能改变类布局（新增 UPROPERTY、改变继承关系等需要完整编译）。
  • 构造函数修改不生效。
  • 新增 .h 文件需要完整编译。
  • 仅 .cpp 中的函数体修改可靠。
