数据结构与算法深度解析 - 从经典理论到游戏开发实战

发表于2026-06-03|更新于2026-06-12|算法与数据结构
|总字数:7.5k|阅读时长:31分钟|浏览量:

引言

游戏开发是数据结构与算法最丰富、最严苛的应用领域之一。每一帧 16ms 的预算内需要完成物理模拟、AI 决策、渲染排序、网络同步、资源加载等数百个系统调度——这迫使我们必须深刻理解数据在内存中的布局、算法的常数因子代价，以及缓存友好的设计哲学。本文从 Unreal Engine 的实际代码出发，系统梳理游戏开发中每一个关键数据结构与算法的选型逻辑和性能权衡。

一、数据结构篇

1.1 数组与链表

动态数组与粒子系统内存预分配

动态数组 ( TArray ) 是 UE 中最常用的容器。其底层采用指数增长的分配策略，默认因子为 1.5 或 2。在粒子系统中，每帧可能生成数千个粒子，频繁的 realloc 会导致严重的内存碎片和 CPU 抖动。

Reserve 背后调用的是 FMemory::Realloc，在大块分配时走的是虚内存映射 (VirtualAlloc)，而非普通的 malloc。对于超过 64KB 的分配，虚内存映射可以减少页表遍历开销。

侵入式链表：TIntrusiveLinkedList

UE 的 Actor 管理大量使用侵入式链表。与 STL std::list 不同，侵入式链表将指针直接嵌入对象内部，消除了额外的节点分配。

性能对比：

操作 std::list TIntrusiveLinkedList
插入 new Node + relink (堆分配) relink only (O(1))
删除 relink + delete Node relink only
遍历 两次指针间接访问 (Node→Data→Field) 一次间接 (Data→Field)
缓存友好度 差 (Node 与 Data 分离) 好 (Data 紧邻下一节点指针)

侵入式链表的劣势是对象只能属于一个链表（同一套指针），多链表场景需要使用 TMultiLinkedList 或额外的侵入式钩子结构。

1.2 栈与队列

命令队列：技能释放缓冲

格斗游戏和 MOBA 中的输入缓冲机制本质是一个定长环形队列。玩家在技能前摇期间输入的指令不会立即执行，而是压入缓冲队列，在前摇结束窗口内依次弹出。

撤销栈在编辑器中的应用

UE 编辑器的事务系统使用双栈结构：Undo 栈 + Redo 栈。每个操作记录为一个 FTransaction 对象，包含序列化的状态快照。

环形队列在音频流处理

音频引擎的混音缓冲使用 lock-free 环形队列 (SPSC Queue)，生产者（音频解码线程）写入 PCM 数据，消费者（音频驱动回调）在中断上下文中读取。

1.3 哈希表

TSet/TMap 与 Robin Hood Hashing

UE 的 TSet 和 TMap 默认使用 Robin Hood Hashing(抢劫哈希)。当插入发生冲突时，比较当前元素与冲突位置元素的探测距离 (Probe Distance)，让”离家更远”的元素占据槽位，以此降低平均探测长度。

性能数据（UE 5.x 内部基准，Intel i9-13900K）：

操作 Robin Hood (UE) std::unordered_map absl::flat_hash_map
插入 10K 0.42ms 0.68ms 0.38ms
查找命中 18ns 24ns 15ns

Robin Hood 的缓存友好性得益于开放寻址——所有数据存储在连续内存中，遍历时指针追逐极少。

FName / FString Table

UE 的 FName 是一个 32 位整数索引，指向全局字符串表中的不可变字符串。字符串表本身使用哈希表，但查找结果被缓存为 FName。

在一个典型的 UE 项目中，FName 比较占了引擎内部比较操作的约 30%——把字符串比较变为整型比较节省了可观的 CPU 时间。

1.4 树结构

四叉树 / 八叉树场景管理

四叉树（2D）和八叉树（3D）是空间划分的经典结构。UE 的 World Partition 系统使用层级网格划分，本质上是一种多分辨率的稀疏八叉树。

BVH 在光线追踪中的应用

包围体层次 (Bounding Volume Hierarchy, BVH) 是光线追踪加速结构的工业标准，替代了早期的 KD 树。UE5 的 Lumen 和硬件光追均使用 BVH。

BVH 构建的核心是对几何图元进行自顶向下或自底向上的分组，使得每个节点的包围体与其子节点包围体的表面积之和最小（SAH - Surface Area Heuristic）。

行为树在 AI 决策

UE 的行为树用于驱动 NPC 的决策逻辑。每个节点根据条件返回 Success / Failure / Running。

行为树的每次 Tick 从根节点向下遍历，执行开销与树深度成正比。因此在深度 > 20 时需要关注子树的黑板缓存机制。

1.5 堆与优先队列

A* 寻路 OpenList

A* 算法的 OpenList 要求快速获取最小 F 值的节点。二叉堆是最常用的实现，但 Fibonacci 堆在 decrease_key 频繁时理论上更优。实践中二叉堆因常数因子小，在 3000 节点以内仍占优。

渲染队列深度排序

半透明物体必须按从远到近的顺序渲染。UE 将半透明物体插入 TArray 后调用 Sort，但 G-Buffer 阶段之前的深度预通道 (Depth Pre-Pass) 已经大幅减少了 Overdraw。

对于粒子系统（数千个半透明四边形），UE 使用桶排序 + 插入排序的组合：按深度离散化为 32-64 个桶，桶内用插入排序。复杂度 O(n + k)，实测比快速排序快 1.5-2x。

1.6 图结构

导航网格的图搜索

NavMesh 是多边形集合，寻路时将其转换为图：每个多边形是一个节点，邻接多边形之间的共享边构成边。

任务依赖图与渲染管线

现代渲染管线是有向无环图 (DAG) 的典型应用。每个 Render Pass（GBuffer、Shadow Map、SSAO、TAA 等）是图的节点，资源依赖（Read/Write）是有向边。

拓扑排序保证每个 Pass 在其所有输入资源就绪后才执行，同时不形成循环依赖（DAG 的必要条件）。

1.7 空间划分结构性能对比

结构 构建时间 查询时间 内存占用 动态更新 最佳场景
均匀网格 O(n) O(1) 单格 中等 易（移动格） 大量均匀分布的小物体
四叉树/八叉树 O(n log n) O(log n) 中等 中等 大小差异大的物体
KD 树 O(n log n) O(log n + k) 高 难（需重建） 静态场景 + 精确查询
BVH O(n log n) O(log n) 高 可 refit 光线追踪、动态场景
空间哈希 O(n) O(1) 平均 低 易 AOI、稀疏分布

UE5 在物理碰撞检测中分层使用：

  • Broad Phase：空间哈希（快速剔除不可能碰撞对）
  • Mid Phase：AABB 树（复合碰撞体内部搜索）
  • Narrow Phase：GJK+EPA（凸体精确检测）

1.8 UE 特有容器选型指南

关键原则：在 UE 中，遍历性能 > 单点插入/删除性能。因为绝大多数游戏逻辑以帧为单位遍历整个容器。

二、算法篇

2.1 排序算法在渲染中的应用

透明物体与深度排序

UE 的 TArray::Sort 底层使用 Introsort（内省排序）：以快速排序开始，当递归深度超过 2 * log₂(n) 时切换为堆排序以规避最坏情况 O(n²)。

粒子排序与基数排序

对 GPU 粒子模拟，CPU 端的排序使用基数排序。基数排序的时间复杂度 O(d*(n+k))，在已知键范围（如 16 位深度值）时非常高效。

基数排序在 UE 中因 SIMD 不支持其访存模式而通常不用于通用排序。但 GPU 上的并行基数排序（如 FidelityFX Sort）能达到 1B keys/s 的吞吐量。

2.2 A* 路径搜索的启发式函数

启发式函数是 A* 算法的灵魂。选择正确的启发式直接影响搜索效率：

Jump Point Search (JPS)：在等权网格上，JPS 跳过多余的邻居展开，只考虑”强制邻居”节点，搜索速度比 A* 快 10-30 倍。不过 JPS 仅限于均匀网格，不适用于 NavMesh 或带权图。

Dijkstra 在开放世界 Streaming

开放世界的关卡流送需要根据玩家位置预加载和卸载地块。这是一个从玩家位置的单源最短路径问题——用加权 Dijkstra，边的权重 = 加载成本。

2.3 拓扑排序与资源加载依赖

UE 的资源加载是典型的拓扑排序场景。材质依赖纹理，蓝图依赖组件类，关卡依赖 Actor 类——形成了有向无环图 (DAG)。

2.5 贪心算法

Frustum Culling 的快速剔除

视锥体裁剪使用分级剔除策略，从粗粒度到细粒度逐步推进，每一步都是一个贪心决策：

  1. 球体测试（O(1)）：包围球在视锥体外 → 直接剔除
  2. AABB 测试（O(1)）：包围盒完全在内部 → 直接通过
  3. 精确相交（O(k)）：在边界上 → 精确测试

LOD 选择的距离权衡

LOD 选择是一个典型的贪心决策：在允许的误差预算内选择最廉价的表示。

2.7 几何算法

凸包 — Graham Scan

物理引擎需要将任意凸体分解为凸包的组合。Graham Scan 是 O(n log n) 的 2D 凸包算法：

GJK + EPA 在 3D 碰撞检测

Gilbert-Johnson-Keerthi (GJK) 算法是 3D 凸体碰撞检测的工业标准。其核心思想是：两个凸体相交当且仅当它们的 Minkowski 差包含原点。

分离轴定理 (SAT)：2D 凸多边形碰撞检测。检查每个边的法向量，若在任一轴上投影不重叠，则两多边形不碰撞。

Delaunay 三角剖分：NavMesh 的生成依赖约束 Delaunay 三角剖分，保证生成的多边形集的最大最小角，避免狭长三角形导致寻路质量下降。

四、实战案例：开放世界 AOI 系统

4.1 需求分析与数据结构选型

AOI (Area of Interest) 系统管理大规模多人在线场景中实体的可见性。1000 个玩家分布在 10km x 10km 的地图上，每个玩家需要获知其周围 200m 范围内的实体变化。O(n²) 的暴力检测不可接受。

选型分析：

候选 查询复杂度 动态更新 适用性
四叉树 O(log n) 需树重构 静态场景好，动态多实体差
KD 树 O(log n) 难更新 不适合
均匀网格 O(1) per cell 极低 内存浪费（空地多）
空间哈希 O(1) 平均 极低 稀疏分布最优

选择：空间哈希 + 九宫格。空间哈希解决稀疏性，九宫格提供直觉化的邻域查询。
