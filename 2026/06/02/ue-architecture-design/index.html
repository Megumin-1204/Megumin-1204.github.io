架构的困境

「混沌から秩序へ」—— 从混沌走向秩序，是每个项目的必经之路。

项目初期，功能少、人少，一切看似美好。当团队扩张到 20+ 人、代码量突破百万行时，如果没有清晰的架构，噩梦就开始了：

  • C++ 和 Lua 的交互散落在各处
  • UI 逻辑、战斗逻辑、网络同步纠缠不清
  • 一个改动引发连锁崩溃

双层架构设计

我们的核心思路：C++ 层提供能力，Lua 层编排逻辑。

协作规范

C++ 开发者的职责

  1. 暴露稳定、最小化的接口给 Lua
  2. UFUNCTION() 标注的方法保持向后兼容
  3. 性能敏感逻辑在 C++ 侧闭环

Lua 开发者的职责

  1. 不直接操作 C++ 内部状态
  2. 复杂逻辑先过事件总线再分发
  3. 热更新时处理好状态迁移

代码评审 Checklist

  • 新增 UFUNCTION 是否必要？
  • Lua 侧是否有对应的错误处理？
  • 跨边界调用频率是否可控？
  • 热更新兼容性验证？

总结

好的架构不在于用了多炫的技术，而在于：

  • 清晰的边界：C++ 和 Lua 各司其职
  • 低耦合：模块间通过事件总线通信
  • 可测试：每个模块可以独立验证

「良いアーキテクチャは、コードに秩序をもたらす」
—— 好的架构，为代码带来秩序。
